# A2450 Log Analysis - 2026-06-09

## Input files

- `device-snapshot.json`
- `a2450-key-events.jsonl`

These files were collected from a Windows machine with an Apple Magic Keyboard-like Bluetooth device connected.

## Device identity

The target keyboard is visible as Apple Bluetooth HID:

```text
BTHENUM\{00001124-0000-1000-8000-00805F9B34FB}_VID&0001004C_PID&029C
```

Windows exposes at least these child HID collections:

```text
HID\{00001124-0000-1000-8000-00805F9B34FB}_VID&0001004C_PID&029C&COL01
HID\{00001124-0000-1000-8000-00805F9B34FB}_VID&0001004C_PID&029C&COL02
```

Observed roles:

- `COL01` = standard keyboard collection, handled by `kbdhid`.
- `COL02` = vendor-defined HID collection, likely where Apple-specific data may live.

## Event statistics

Parsed event log:

```text
Total events: 1196
Events from Apple PID 029C COL01: 1115
Events from other devices: 75
Events with null device name: 6
Events with RawReportHex: 0
```

The current Raw Input logger only captured keyboard-style events from `COL01`.

## Key findings

### 1. Fn / Globe is not visible as a normal Raw Input keyboard key

No clear Fn-specific virtual key, scan code, or RawReportHex appeared in the `COL01` keyboard event stream.

This means the current user-mode Raw Input logger is not enough to prove Fn state.

### 2. F1-F12 are visible as normal function keys

Observed standard F-row virtual keys:

```text
F1  = VK 112, ScanCode 59
F2  = VK 113, ScanCode 60
F3  = VK 114, ScanCode 61
F4  = VK 115, ScanCode 62
F5  = VK 116, ScanCode 63
F6  = VK 117, ScanCode 64
F7  = VK 118, ScanCode 65
F8  = VK 119, ScanCode 66
F9  = VK 120, ScanCode 67
F10 = VK 121, ScanCode 68
F11 = VK 122, ScanCode 87
F12 = VK 123, ScanCode 88
```

### 3. Command keys are visible as Windows keys

Observed:

```text
Left Command  -> VK 91, ScanCode 91
Right Command -> VK 92, ScanCode 92
```

### 4. Option is visible as Alt

Observed:

```text
Option -> VK 18, ScanCode 56
```

The current logger does not yet distinguish left/right Option cleanly.

### 5. Ctrl is visible as Ctrl

Observed:

```text
Ctrl -> VK 17, ScanCode 29
```

The current logger does not yet distinguish left/right Ctrl cleanly.

### 6. Fn-layer navigation was not visible through standard Raw Input

The parsed event log did not show standard `Delete`, `Home`, `End`, `PageUp`, or `PageDown` virtual keys from Apple PID 029C COL01.

This suggests one of these is true:

1. Fn was not captured by the current logger.
2. Fn-layer transformations are not emitted through Windows Raw Input.
3. Fn state is carried through the vendor-defined HID collection `COL02`.

## Current conclusion

The keyboard is correctly detected as Apple Bluetooth HID with PID `029C`, but the first Raw Input logger is insufficient for the core Fn/Ctrl swap goal.

The next development step should not be a driver yet. The next step should be a lower-level HID report logger for the Apple `COL02` vendor-defined collection.

## Next recommended implementation

Add a second logger:

```text
tools/A2450RawHidLogger
```

Purpose:

- Enumerate HID devices containing `VID&0001004C_PID&029C`.
- Identify `COL02` vendor-defined collection.
- Open the device with read access.
- Log raw input reports while pressing:
  - Fn / Globe
  - Fn + Left Ctrl
  - Fn + F1-F12
  - Fn + Backspace
  - Fn + Arrow keys
- Write output to:

```text
logs/a2450-raw-hid-reports.jsonl
```

## Decision after Raw HID logging

If `COL02` exposes Fn state:

- Build a user-mode MVP:
  - Raw HID reader detects physical Fn.
  - Low-level keyboard hook intercepts physical Left Ctrl.
  - SendInput synthesizes Left Ctrl when Fn is pressed.
  - Physical Left Ctrl becomes an internal FnLayer state.

If `COL02` does not expose Fn state:

- Move to KMDF / HIDClass lower-filter driver research.
